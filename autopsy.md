# Autopsy

Autopsy is an open-source digital forensics platform used to examine disk images, file systems, extracted artifacts, and investigation findings. It is useful for structured forensic analysis because it guides the analyst through case creation, data-source selection, ingest processing, artifact review, and reporting.

## Basic Workflow

Use this workflow as the normal sequence for an Autopsy investigation:

1. Create or open a case for the data source being investigated.
2. Select the data source to analyze.
3. Configure ingest modules to extract specific artifacts from the data source.
4. Review artifacts produced by the ingest modules.
5. Generate a report that preserves findings for later review.

## Case Analysis

- Autopsy case files use the `.aut` file extension.
- A case is the investigation container. It keeps the configured data sources, ingest results, tags, comments, and reports together.

## Data Sources

Autopsy can analyze multiple disk image formats.

### Supported Disk Image Formats

- **Raw single image:** `.img`, `.dd`, `.raw`, `.bin`
- **Raw split image:** `.001`, `.002`, `.aa`, `.ab`, and similar split formats
- **EnCase image:** `.e01`, `.e02`, and related EnCase evidence files
- **Virtual machine images:** `.vmdk`, `.vhd`

When an image is split into multiple parts, such as `E01`, `E02`, and `E03`, point Autopsy to the first image file. Autopsy will handle the remaining parts.

## Ingest Modules

Ingest modules are Autopsy plug-ins that analyze a selected data source and extract specific artifact types.

- They can be configured during the source-addition stage or later from the dashboard.
- By default, ingest modules are configured to run on **all files, directories, and unallocated space**.
- Module output populates the **Results** node in the Tree Viewer.
- Some modules have per-run settings, so review configuration before launching analysis.

## Tree Viewer

The Tree Viewer has five top-level nodes:

| Node | Purpose |
|---|---|
| Data Sources | Organizes source data similarly to Windows File Explorer. |
| Views | Organizes files by file type, MIME type, file size, and similar views. |
| Results | Displays results produced by ingest modules. |
| Tags | Displays files or results tagged by an ingest module or analyst. |
| Reports | Displays reports generated by modules or by the analyst. |

## Result Viewer

Do not confuse the **Results** node in the Tree Viewer with the **Result Viewer** pane.

The Result Viewer has three tabs:

- **Table**
- **Thumbnail**
- **Summary**

Use the Result Viewer to inspect artifacts extracted by ingest modules.

## Contents Viewer

The Contents Viewer includes columns that may not be immediately obvious.

| Column | Meaning | Operational Use |
|---|---|---|
| S = Score | Shows whether the item has been marked as notable or suspicious. | A red exclamation point indicates notable content. A yellow downward triangle indicates suspicious content. |
| C = Comment | Shows whether the file or folder has an analyst comment. | A yellow page icon indicates that a comment exists. |
| O = Occurrence | Shows how often the file or folder has appeared in past cases. | Requires the Central Repository. Useful for identifying known or previously seen artifacts. |

## Data Sources Summary

The **Data Sources Summary** provides summarized information across nine categories. Treat it as an overview of total findings, not a substitute for artifact-level review.

For deeper analysis, examine each module's output through the Result Viewer.

## Generate Report

Autopsy can create reports in multiple formats.

Reports are useful because they:

- Preserve investigation findings after the live analysis ends.
- Provide data sheets for forensic cases.
- Include information listed under the Result Viewer pane.
- Support re-investigation or follow-on review.

Reports have an important limitation: they do not provide additional search options. If an artifact is needed later, the analyst may need to manually locate it in the report.

# Acquiring a Digital Forensic Image

Forensic image acquisition is the process of collecting data from a device in a way that preserves evidence integrity and supports later analysis.

## Acquisition Types

| Acquisition Type | Description | Use Case |
|---|---|---|
| Static acquisition | Creates a bit-by-bit image while the device is powered off. | Best when the disk is not encrypted and volatile data is not required. |
| Live acquisition | Creates a bit-by-bit image while the device is powered on. | Useful when encryption, active sessions, or RAM contents matter. |
| Logical acquisition | Copies a selected list of files from the device. | Faster and more targeted, but less complete than full imaging. |
| Sparse acquisition | Copies selected fragments of unallocated data. | Useful for limited deleted-data collection, but not as complete as static or live acquisition. |

## A Computer That's Switched Off

If the evidence is a powered-off Windows 10 laptop and the disk is not encrypted, avoid turning it on. Booting the device can modify disk contents and tamper with evidence.

A basic static acquisition workflow is:

1. Remove the hard drive or SSD from the laptop.
2. Use a hardware write blocker to prevent changes to the original media.
3. Use forensic imaging software to create a raw image or equivalent bit-for-bit copy.
4. Save the image to suitable evidence storage.

## A Computer That's Switched On

If the device is already powered on, do not automatically shut it down.

A live acquisition may be necessary because:

- The disk may be encrypted, and shutdown could make data unreadable without the password or recovery key.
- Volatile memory may contain important evidence.
- Active sessions may give investigators access to logged-in accounts or services.

Live acquisition tools usually require executing a program on the target system to collect volatile and non-volatile data.

## Acquiring a Smartphone Image

Smartphones are common digital evidence sources, but modern phones are usually encrypted by default.

Key points:

- Encrypted storage appears as random data without the decryption key.
- Access to the decryption key or an unlocked device is often essential for analysis.
- Mobile acquisition can be more constrained than traditional computer acquisition because of platform security controls.

## Encryption in Smart Phones

### Android Encryption

- **Android 4.4** introduced full-disk encryption. When active, user-created data is encrypted before being written to storage and decrypted before being read. The device cannot fully boot without the password.
- Full-disk encryption applies to built-in storage and does not automatically include removable storage such as microSD cards.
- **Android 7.0** introduced Direct Boot and file-based encryption. File-based encryption allows different files to use different keys, enabling limited basic functionality before full user unlock.
- Depending on settings and Android version, an SD card might also be encrypted. Android 9.0 and later can encrypt SD cards similarly to internal storage.
- Since **Android 6.0**, encryption has been mandatory. Unless the phone is very old, expect Android devices to be encrypted.

### iPhone Encryption

- Apple iPhone devices are encrypted by default.
- iOS uses Data Protection, a file-based encryption methodology.

## Android Backup with ADB

McBlue uses Android Debug Bridge (`adb`) and Autopsy for Android analysis.

Once the phone is unlocked and connected to the analysis laptop, a logical backup can be created with:

```bash
adb backup -all -f android_backup.ab
```

Command meaning:

- `backup -all` backs up all applications that permit backup.
- `-f android_backup.ab` saves the backup to `android_backup.ab`.

Important limitation: some applications explicitly disable backup with `allowBackup=false`. Also, `adb backup` has been restricted since Android 12 and should not be treated as a complete acquisition method.

## Raw Android Image Acquisition

A logical backup is useful, but McBlue wants a full raw image of the phone storage.

Commercial mobile forensics products often rely on `adb` combined with additional methods, such as an exploit to gain root access. Root access is needed because Android does not normally give users full access to all files, directories, or raw disk devices.

After obtaining root access, McBlue confirms identity and locates the device mounted at `/data`:

```bash
whoami
mount | grep data
```

![Practical Case - Android root access and data partition identification](/images/autopsy-101.png)

The output identifies `/dev/block/dm-0` as the device mounted on `/data`. This can be treated as a partition on the smartphone's storage.

To pull the raw device image:

```bash
adb pull /dev/block/dm-0 Android-McGreedy.img
```

This command saves `/dev/block/dm-0` to the local file `Android-McGreedy.img`. Root access is required. In this case, the resulting image is approximately 6 GB.
